❯ ISR 是什么,SSR 是什么?                                                        

  Thought for 2s (ctrl+o to expand)

⏺ 两个都是 Next.js 的页面渲染策略：

  ---
  SSR — Server-Side Rendering（服务端渲染）

  每次请求都在服务器上实时生成页面，然后返回完整 HTML 给浏览器。

  用户访问 → 服务器现做页面 → 返回 HTML → 浏览器展示

  适合内容经常变、需要个性化的场景。比如你登录后的
  Dashboard，不同用户看到不同数据，每次请求都要最新。

  ---
  ISR — Incremental Static Regeneration（增量静态再生成）

  预先生成静态页面，但可以按时间自动刷新，也可以按需重新生成。

  首次：服务器预做页面存起来 → 用户访问直接拿缓存 → 秒开
  5分钟后有新请求：如果缓存过期，后台重新生成新页面，下次请求就是新的

  适合内容偶尔变、但大量用户看到的都一样。比如你的法律页面（/privacy），内容基本
  不改，不需要每次请求都生成一遍。

  ---
  和你的项目什么关系？

  你的项目其实天然避开了这些限制：

  - 首页是纯交互式的（上传拖拽、定价表），不需要服务端渲染数据 → 客户端组件搞定
  - 法律页面是纯静态文字 → 构建时生成一次就够了（SSG，比 ISR 更简单）
  - API 用的是 Edge Route，走 Worker 运行时，不涉及页面渲染

  所以不管你选 A 还是 B，ISR/SSR 的限制都不会影响你的 MVP。

  ---
  建议：Cookie + IP 组合

  两个维度同时记，但角色不同：

  Cookie  → 标识"这个人"（谁传的）
  IP      → 做硬上限兜底（不管谁，这个 IP 今天最多传 N 次）

  1. 访客首次上传时，设一个Cookie（UUID），写入 D1 时带 cookie_id 字段

    - 这个 Cookie 今天传了多少次 ← 主要限制（10次）
    - 这个 IP 今天总共传了多少次 ← 安全兜底（比如 50 次硬顶）

  ┌───────────────────────┬────────────────────────────────────────────────┐
  │         场景          │                      结果                      │
  ├───────────────────────┼────────────────────────────────────────────────┤
  │ 正常人用              │ Cookie 追踪，10次到了就停 ✅                   │
  ├───────────────────────┼────────────────────────────────────────────────┤
  │ 想清 Cookie 刷次数    │ IP 硬顶兜底，总能拦住 ✅                       │
  ├───────────────────────┼────────────────────────────────────────────────┤
  │ 办公室多人用同一 IP   │ 各自 Cookie 独立计数，不影响别人 ✅            │
  ├───────────────────────┼────────────────────────────────────────────────┤
  │ 换了 WiFi + 清 Cookie │ 彻底新身份，没法挡（但这成本太高，几乎没人做） │
  └───────────────────────┴────────────────────────────────────────────────┘

     Step 5: 核心上传 API（POST /api/upload）

     ▎ 这是整个应用的核心代码，需要最细致的实现

     文件：app/api/upload/route.ts（Edge Runtime）

     执行流程：

     1. 接收 Request → 提取 FormData(file, cf-turnstile-response)
     2. Cookie 检查 → 读取/设置 cookie_id（UUID），用户未登录时用此字段追踪
     3. 上传次数检查：
        a. D1 查询：cookie_id 今日上传次数
        b. D1 查询：IP 今日总上传次数
        c. 次数 >= 5 → 校验 Turnstile token（调用 CF API）
        d. 次数 >= 10（cookie_id）或 >= 50（IP）→ 429 Too Many Requests
     4. 文件校验：
        a. MIME Type：白名单 image/jpeg, image/png, image/webp, image/gif
        b. Magic Bytes：读文件头 4-8 字节，验证真实格式（防 MIME 冒充）
        c. 文件大小：按 plan_type 限额校验
     5. 存储限额检查：
        a. 登录用户：D1 查询 users.storage_used + 当前文件大小 <= 限额
        b. 访客用户：D1 查询 cookie_id 下所有文件总大小 <= 200MB
     6. R2 上传：
        a. Key 格式：uploads/{YYYY}/{MM}/{UUID}.{ext}
        b. 使用 R2 binding 直接上传
     7. D1 写入 files 表：
        a. user_id（可为 NULL）、cookie_id、r2_key、file_size、mime_type
        b. expires_at = NOW + 30 days（Free/Guest），NULL（Plus/Enterprise）
        c. 更新 users.storage_used（登录用户）
     8. 更新计数：
        a. 自增 cookie_id 今日上传计数
        b. 自增 IP 今日上传计数
     9. 返回 JSON：
        {
          "success": true,
          "url":        "https://cdn.photo2url.com/uploads/2026/07/{UUID}.jpg",
          "preview":    "https://photo2url.com/i/{UUID}",
          "markdown":
     "![image](https://cdn.photo2url.com/uploads/2026/07/{UUID}.jpg)",
          "html":       "<img
     src=\"https://cdn.photo2url.com/uploads/2026/07/{UUID}.jpg\" />"
        }

     辅助文件：
     - utils/magic-bytes.ts — Magic Bytes 校验表（JPEG: FF D8、PNG: 89 50 4E
     47、GIF: 47 49 46 38、WEBP: 52 49 46 46）
     - utils/upload-limit.ts — 限额常量 + 判断函数
     - utils/id-generator.ts — UUID 生成

     ---
     Step 6: 上传计数器表（D1 新增）

     为支持 Cookie + IP 计数，在 D1 加一张轻量表：

     CREATE TABLE upload_counts (
         id INTEGER PRIMARY KEY AUTOINCREMENT,
         cookie_id TEXT,        -- 访客 Cookie ID（可不唯一）
         ip TEXT,               -- 上传者 IP
         upload_date TEXT NOT NULL,  -- 'YYYY-MM-DD'
         count INTEGER DEFAULT 1,
         UNIQUE(cookie_id, ip, upload_date)
     );

     每次上传成功后 INSERT OR UPDATE count = count + 1。

     端到端测试场景

     1. 访客上传流程
       - 打开首页 → 拖拽一张 JPG → 结果卡片出现 → 复制 Markdown → 粘贴到 Notion
     → 图片正常显示
       - 连续上传 6 次 → 第 6 次出现 Turnstile 验证
       - 上传超过 10 次 → 返回 429
     2. 注册/登录流程
       - 点 Login → Google 登录 → 回到首页 → 导航栏显示头像
       - 点 Dashboard → 看到空文件列表
     3. 付费流程
       - 点 "Plus 订阅" → Lemon Squeezy 结账弹窗 → 完成支付
       - 返回 Dashboard → 看到 plan_type 已更新为 Plus
     4. 图片预览页
       - 打开 photo2url.com/i/{UUID} → 看到图片 + 复制按钮
     5. 过期清理
       - 手动将一条 files 记录 expires_at 设为过去时间
       - 触发 Cron → 确认文件从 R2 和 D1 中删除

     技术验证

     - wrangler pages dev 本地运行全栈
     - 上传文件确认 R2 落盘
